# macOS 长篇叙事与语速控制评估

> 状态：研究记录，尚未形成产品决策，也不授权实现。  
> 评估日期：2026-07-21  
> 范围：macOS；iOS 明确暂缓。  
> 隐私：本文只记录通用需求，不记录私人音色名称、文稿内容或本地文件路径。

## 先读结论

1. **Vocello 确实基于 Qwen3-TTS。** 项目声明的官方模型来源是
   [QwenLM/Qwen3-TTS](https://github.com/QwenLM/Qwen3-TTS)，Apple 端运行时沿用了并已内化修改
   `mlx-audio-swift`，模型权重来自 `mlx-community` 的 Qwen3-TTS 1.7B 4-bit/8-bit 版本。
2. **13,000–18,000 字可以做成“一次提交、最终得到一个音频”的产品体验，但不应解释为一次模型推理。**
   可靠实现必须在内部规划分段、逐段生成、校验、断点续作并有界合并。Windows 修改版所谓“长文本直出”
   很可能也是封装了这条流水线；取得其确切源码前不能断言实现细节。
3. **项目不需要从零重做长篇基础。** 当前出货路径能生成分段文件和 schema-v3 清单；另外已有尚未接入产品的
   schema-v4 中文分段规划器和有界 WAV 合并器。优先候选是完成这两套基础到产品流程的切换，而不是另写一套
   临时切句与拼接逻辑。
4. **对克隆音色提供可重复的 `1.00 / 0.90 / 0.85` 数值语速，首选评估“保持音调的离线时间伸缩”。**
   Qwen3-TTS 的自然语言指令适合表达风格，但不是稳定的数值速率接口，而且 Base/Clone 路径并不具备相同的
   指令控制能力。直接改 WAV 采样率会同时降低音调，不适合作为正式实现。
5. **Apple 原生 `AVAudioUnitTimePitch` + 离线渲染应作为第一项技术验证，但还不是最终选型。** 是否逐段变速、
   是否让段间停顿也随之变长、长音频上的音质与吞吐，都必须用目标叙事音色做 A/B 后再决定。
6. **不能直接宣称 MLX 与 CUDA“持平”。** 两者都能硬件加速，但跨机器、跨量化和跨实现的结果不可比。
   应先统一文本、模型档位、热机状态和统计口径，再分别记录模型生成、变速、合并与总耗时。

## 需求边界

本次评估对应的产品目标是：

- 接收约 13,000–18,000 个中文字符的助眠叙事文稿。
- 用户只提交一次任务，最终得到一个连续、可播放、可进入历史记录的 WAV。
- 支持至少 `1.00`、`0.90`、`0.85` 三档可复现语速，重点覆盖保存的克隆音色。
- 语速降低时尽量保持原音高、音色身份和自然韵律。
- 任一分段失败或应用重启后，不必从第一段全部重做。
- 长任务内存应随“当前处理块”而不是随总音频时长增长。
- 全程本地处理；原稿、音色和中间音频不进入公开测试证据。
- 本轮不设计或实现 iOS 版本，也不把 CUDA 移植到 Apple 平台。

## 已确认的项目现状

| 项目 | 当前事实 | 对本需求的含义 |
| --- | --- | --- |
| 模型谱系 | `QwenVoiceBackendProvenance` 指向 Qwen3-TTS；生产目录使用 1.7B 4-bit Speed 和 8-bit Quality 权重 | 可以确认模型家族，不代表与官方 Python/CUDA 或未知 Windows 分支逐行相同 |
| 当前长文本入口 | 超过产品阈值后进入 `LongTextGenerationRouter`、`LongFormBatchSegmenter`、`BatchGenerationRunner` | 已有批量生成能力，但结果仍是独立分段 WAV + manifest v3 |
| 当前分段器 | 默认每段约 900 字；句末识别仅覆盖 `. ! ?`，超长内容再按空白拆分 | 对缺少空格的中文长句不够稳健，不宜直接扩展成最终方案 |
| v4 分段基础 | `SpokenTextPlanning.swift`、`LongFormPlanning.swift` 已支持中文标点、受保护范围、稳定段 ID、子种子、停顿与版本化身份 | 应复用为正式长篇规划基础，但目前还不是出货路径 |
| 有界合并基础 | `BoundedLongFormAssembler` 能以固定音频块分析并原子写出 24 kHz 单声道 PCM16 WAV | 已解决“合并时内存随全长增长”的核心问题，但尚未接进产品协调器 |
| 语速字段 | History 的 `Generation.speed` 已存在，但现有生成路径写入 `nil` | 数据模型预留不等于已经实现语速处理 |
| 语速处理器 | 产品代码中没有正式的 TimePitch/tempo 输出阶段 | 需要独立验证、质量合同和失败恢复语义，不能只加一个 UI 滑块 |

更完整的现状边界见[长文本生成参考](long-form-generation.md)和
[运行时收敛决策](../decisions/runtime-streaming-quality-convergence.md)。当前进度仍将 v4 长篇标记为基础阶段，
不得因类型已经存在就宣称功能完成。

## 为什么不能把 18,000 字交给一次推理

Qwen3-TTS 12Hz 模型在本项目中的编解码帧率实际为 12.5 帧/秒。当前产品生成策略的
`maxNewTokens` 是 2,048，模型检查点默认上限是 8,192。仅按理论上限换算：

| 最大 codec token | 理论音频上限 | 说明 |
| ---: | ---: | --- |
| 2,048 | 163.84 秒 | 当前产品策略，约 2 分 44 秒 |
| 8,192 | 655.36 秒 | 检查点默认上限，约 10 分 55 秒 |

13,000–18,000 字的助眠稿通常远长于上述单次音频窗口。单纯抬高 token 上限还会扩大 KV cache、失败重做成本、
随机漂移和尾段截断风险。因此，“长文本直出”应定义为**一个用户任务和一个最终产物**，而不是一个模型调用。

## 长篇方案比较

| 方案 | 优点 | 主要问题 | 评估 |
| --- | --- | --- | --- |
| 提高 token 上限，单次推理 | 表面流程简单 | 无法覆盖完整稿件；内存、截断、失败重做和音色漂移风险集中 | 排除 |
| 继续扩展当前 v3 切句并直接拼接 | 改动较小 | 中文边界、稳定身份、恢复、最终原子产物和质量证据不足 | 只可作为临时对照，不建议成为正式架构 |
| 接入 v4 规划器 + 逐段生成 + 有界合并 | 中文边界更稳；可恢复；内存有界；已有测试基础 | 仍需产品协调器、逐段 QA、History 和恢复策略 | 首选候选 |
| 调用独立外部脚本完成全部工作 | 可快速复现实验 | 与应用下载、历史、取消、签名和分发割裂 | 可做研究对照，不作为首选产品路径 |

### 首选候选流程

```text
原始文稿
  -> 口语文本规划（保留私密原文，只发布摘要身份）
  -> v4 长篇分段计划（稳定段 ID、子种子、停顿、版本）
  -> 顺序生成单段原始 WAV
  -> 单段快速质量检查
  -> 可选：保持音调的语速处理
  -> 变速后信号检查
  -> 有界合并（边缘处理 + 显式停顿 + 原子写出）
  -> 最终文件可读性、帧数、连续性和摘要校验
  -> History 中只接受一个最终项目
```

首版应保持顺序生成。并行段生成会提高峰值内存、模型状态竞争和输出漂移风险，不适合在长篇基础尚未完成产品切换时
同时引入。

## 语速控制方案比较

### 1. 让模型“说慢一点”

Qwen3-TTS 的 CustomVoice/VoiceDesign 支持自然语言指令，可用于“从容”“舒缓”等表达风格；这不等于精确的
`0.85x`。对相同文字，实际时长还会受停顿、采样随机性和韵律影响。更重要的是，保存音色克隆使用 Base/Clone
能力，不能假设拥有同等指令控制。

结论：可以保留为风格控制或未来实验，不能承担克隆音色的精确数值语速合同。

### 2. 改采样率或普通 Varispeed

它实现简单，但时间和音高一起变化。若只改变播放/解释采样率：

- `0.90x` 约降低 `1.82` 个半音；
- `0.85x` 约降低 `2.81` 个半音。

这会让声音明显变沉，削弱克隆身份一致性。结论：只适合故障诊断或刻意变声，不作为正式方案。

### 3. Apple `AVAudioUnitTimePitch` 离线渲染

Apple 提供独立的 rate 与 pitch 控制，并可通过 `AVAudioEngine` 的离线手动渲染把结果写成文件。它不增加独立
可执行文件或第三方许可证，最适合先做 macOS 原型验证。

尚需实测的问题包括：

- 30–120 分钟素材上的离线吞吐、内存和确定性；
- 男声低频、气声、齿音、颤音和长元音在 `0.90/0.85` 下是否出现相位或颗粒感；
- 每段重置处理状态与整条连续处理之间的接缝差异；
- 取消、失败、临时文件和原子替换语义。

结论：**第一技术验证候选，不等于已经选定。**

### 4. FFmpeg `atempo`

官方 `atempo` 支持音高相对稳定的 tempo 调整，适合作为质量和速度对照。但把 FFmpeg 嵌入签名应用会新增二进制
体积、分发、沙盒/进程管理和许可证审计工作。

结论：适合实验基线；只有 Apple 原生方案未达质量门槛时才评估进入产品。

### 5. Rubber Band

Rubber Band 是成熟的时间伸缩/移调库，但开源版采用 GPL，闭源分发通常需要商业许可证。

结论：可作为高质量对照或商业备选，不应在许可证决策前直接引入。

## 变速放在哪个阶段

这是尚未决定的关键架构点。

| 位置 | 优点 | 代价 | 停顿语义 |
| --- | --- | --- | --- |
| 每段生成并检查后、合并前 | 内存天然有界；单段可重试；失败不用重做全片；处理器重置发生在自然边界 | 必须验证段间音色和相位接缝；多次初始化 | v4 计划中的显式段间停顿保持原时长 |
| 最终 WAV 合并后统一处理 | TimePitch 状态连续，理论上减少处理器重置痕迹 | 需要再次完整读写；失败重做整条；临时磁盘更大 | 人声和已经插入的停顿一起按比例拉长 |

对助眠叙事，本文目前**倾向先验证逐段变速再合并**：它与现有有界处理和恢复模型更一致，也允许“人声慢下来但
计划停顿不被机械放大”。不过这仍需实际音色 A/B；如果接缝质量不达标，应测试最终成品统一处理。

## `0.90` 和 `0.85` 的产品含义

在保持文字内容不变时：

- `0.90x` 使被处理的人声时长增加约 `11.11%`；
- `0.85x` 使被处理的人声时长增加约 `17.65%`。

需要由产品体验而不是代码擅自决定：

1. 数值只作用于发声部分，还是作用于整条节目（包含段间静音）？
2. 句内由模型生成的停顿会随音频一起伸长；v4 合并器额外插入的段间停顿是否保持固定？
3. `0.85` 是“精确输出时长比例”，还是用户可感知的“慢速档”名称？
4. 新任务默认 `1.00`，还是记忆上一次选择？现阶段更安全的默认值是 `1.00`，避免静默改变已有输出。

## 质量验收建议

在 UI 开发前，先做无 UI、可重复的文件级实验。至少包含一组代表性的慢节奏男声克隆素材，但测试记录不得暴露
私人音色名称或文稿。

| 检查项 | 建议门槛/观察 |
| --- | --- |
| `1.00x` 旁路 | 不经过 TimePitch；输出不得因“启用语速功能”而无意义重编码 |
| 时长比例 | `0.90/0.85` 的发声区间时长应落在预定容差内；容差在原型数据后冻结 |
| 文件合同 | 24 kHz、单声道、PCM16、帧数与摘要可验证，临时文件不冒充最终结果 |
| 信号安全 | 无削波、异常 DC、突发静音、明显点击或超限段边界跳变 |
| 内容保真 | 分段做适用的 ASR/文本一致性检查，避免整本音频一次识别造成不可定位失败 |
| 音色与韵律 | 比较身份相似度、基频轨迹、语速分布、停顿分布；人工试听只作为产品偏好补充 |
| 恢复 | 中途取消、应用重启、单段失败后只重做身份不匹配或失败的段 |
| 有界资源 | 1、10、100 段验证稳态内存不随总时长线性增长，无 critical pressure/hard trim/full unload |

建议的 A/B 素材分两层：

- 3–5 分钟片段：快速比较 `1.00/0.90/0.85`、逐段/整条以及 Apple/对照算法。
- 30–120 分钟长篇：验证接缝、累计漂移、离线吞吐、磁盘占用、取消和恢复。

素材应覆盖中文句号、问号、感叹号、引号、数字、日期、缩写、长段落和少量对白。不能只用短英文句子验证中文长篇。

## 性能比较口径

CUDA 和 MLX 只能在同一任务合同下分别报告，不能用营销页或一次短句结果互相换算。每次记录至少拆成：

- 冷启动模型加载时间与热机生成时间；
- 模型/量化档位（Speed 4-bit 或 Quality 8-bit）；
- 输入字符数、计划段数、最终音频秒数；
- 模型生成耗时、TimePitch 耗时、合并/校验耗时、总墙钟时间；
- 峰值应用内存、引擎内存和临时磁盘占用；
- 明确定义的实时系数，避免把 `音频秒/墙钟秒` 与其倒数混写。

未知 Windows 修改版必须先取得确切源码或可运行目录，确认其模型、量化、切段、并发、拼接、语速算法和计时范围，
再做同稿对比。未经审计不能复制其二进制、依赖或实现到 Vocello。

## 工作区、恢复与 History 语义

长任务应拥有私有工作区和版本化清单。当前语速处理已经暂缓，因此同一规划分段只允许保留一份已接受 WAV，
不创建原始版和变速版重复副本。建议原则：

- 计划身份绑定规范化规则、原文摘要、模型工件、音色条件、采样策略、子种子、目标语速与处理算法版本。
- 只有上述身份匹配的已接受段才能复用；更换音色、模型、语速或算法不得误用旧缓存。
- 合并过程写临时目标，校验成功后原子发布；失败文件不进入 History。
- History 默认只展示一个最终音频，分段是项目恢复证据，不是几十条普通生成记录。
- 最终 WAV 和 History 接受前不删除唯一可恢复的分段；两者接受后立即删除分段 WAV、失败/替换片段和临时合并文件。
- 可恢复失败只暂存与当前计划身份匹配的已接受分段；用户放弃/取消任务时删除整个任务工作区。
- 应用启动时清扫已完成、已放弃或身份不匹配的孤儿工作区；清理失败只保留最小 `cleanupPending` 状态并重试。
- 成功任务只保留最终 WAV 和必要的小型摘要元数据，不保留可播放的过程音频。
- 公开日志和基准只记录摘要、计数、版本与质量结果，不记录原稿、音色名称、本地路径或音频。

## 建议的分阶段工作（不是排期承诺）

### 阶段 0：取得对照证据

- 审计 Windows 修改版的确切来源与实现，不依据界面印象推断。
- 准备隐私安全的代表性中文片段与长篇测试集。
- 明确“变慢是否包含段间停顿”的听感目标。

### 阶段 1：独立语速原型

- 不改 UI，先用 Apple 原生离线 TimePitch 处理已有 WAV。
- 比较逐段与整条处理，必要时加入 FFmpeg/Rubber Band 作为外部对照。
- 冻结质量、时长、性能和失败处理合同后再决定算法。

### 阶段 2：macOS 长篇 `1.00x` 产品切换

- 把 v4 规划器、顺序生成、逐段 QA、有界合并、恢复和单一 History 产物接成一条路径。
- 先证明不变速长篇可靠，避免把分段、合并和 TimePitch 三类问题混在一起调试。

### 阶段 3：整合语速

- 在已验证的阶段位置接入目标语速，持久化实际算法与速度身份。
- 验证 `1.00/0.90/0.85`、取消/恢复、磁盘和长时稳定性。

### 阶段 4：产品界面与 macOS 验收

- 最后再增加语速控件、进度、恢复提示、磁盘提示与 History 展示。
- 前端验收按项目规定使用 macOS XCUITest；不把 iOS 混进本轮。

## 主要风险

| 风险 | 缓解方向 |
| --- | --- |
| 中文切段破坏语义或数字/URL | 复用 v4 受保护范围与边界优先级，建立中文固定样本 |
| 长篇音色或情绪逐段漂移 | 稳定条件身份与子种子；按段记录质量结果；用代表性音色做长时比较 |
| TimePitch 产生颗粒、相位或齿音伪影 | 先做 Apple 原生 A/B，保留外部高质量算法作对照，不以“能跑”代替质量门槛 |
| 段边界点击、重复或停顿不自然 | 复用有界合并器的边缘分析、淡化、停顿和边界跳变证据 |
| 中途失败导致数小时重做 | 稳定段 ID、原子段产物、身份匹配恢复和显式替换 lineage |
| 中间 WAV 占用大量磁盘 | 启动前估算、展示空间、最终接受后按策略清理，不在失败前提前删除证据 |
| 第三方算法妨碍签名/分发 | 优先验证系统框架；FFmpeg/Rubber Band 进入产品前单独完成许可证和打包审计 |
| 同时修改长篇、语速、UI 导致难以归因 | 按“语速原型 -> 1.00 长篇 -> 语速整合 -> UI”分阶段验证 |

## 明确暂缓的决策

在得到实际 A/B 和 Windows 实现证据前，以下内容不应写成既定方案：

- Apple TimePitch 是否达到最终音质，以及是否需要 FFmpeg 或商业算法。
- 逐段变速还是最终成品统一变速。
- 段间显式停顿是否随 `0.90/0.85` 缩放。
- 每段目标长度、停顿时长与时长比例容差的最终数值。
- 中间段在成功后保留多久，是否向普通用户显示。
- 默认语速、是否记忆上次语速、Speed/Quality 的默认组合。
- 是否需要导入 Windows 分支的任何实现；在源码审计前答案是“不导入”。

## 本轮非目标

- 不实现 iOS 长篇或语速功能。
- 不承诺“一次模型调用生成 18,000 字”。
- 不把 CUDA 移植到 macOS，也不宣称 CUDA/MLX 已经性能持平。
- 不做模型微调或重新训练。
- 不在原型证据前增加产品 UI 或引入第三方二进制。
