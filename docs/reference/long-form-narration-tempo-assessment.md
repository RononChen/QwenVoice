# macOS 长篇叙事与语速控制评估

> 状态：macOS 已接入 200 字级分段、固定段间停顿、完整成品 FFmpeg `atempo` 变速和最小 LGPL-only 正式发布组件；Developer ID 公证候选版仍待下一次正式发布执行。
> 评估日期：2026-07-22
> 范围：仅 macOS；iOS 明确暂缓。
> 隐私：本文只记录通用结论，不记录私人音色名称、文稿内容或本地文件路径。

## 结论

1. Vocello 使用 Qwen3-TTS 模型家族，Apple 端通过 MLX 运行。
2. 13,000–18,000 字的“长文本直出”应定义为：用户一次提交，应用内部自动分段、顺序生成和合并，最终只得到一个 WAV；它不是一次模型推理。
3. 对 Windows 修改版的独立静态分析与同稿波形分析相互印证了关键行为：约 200 字分段、相邻分段间插入 0.3 秒静音、合并后对完整音频执行一次 FFmpeg `atempo`。
4. 不导入 Windows 程序、Cython 二进制或未知依赖，只复刻已经由外部行为证明的通用算法和产品语义。
5. Apple `AVAudioUnitTimePitch` 已在同源 A/B 中被否决：与 FFmpeg `atempo` 相比，其包络、基频和频谱包络偏差明显更大。macOS 开发路径改用 FFmpeg `atempo`。
6. 正式 macOS 包不打包 Homebrew/Windows 的完整 GPL 构建；现在从固定 FFmpeg 8.0.3 官方源码构建、审计并单独签名最小 LGPL-only `ffmpeg-vocello` helper。普通开发构建仍可回退到本机 FFmpeg。
7. CUDA 与 MLX 都能提供硬件加速，但在机器、量化、热机状态和统计口径统一前，不能宣称两者性能持平。

## 产品目标

- 接收约 13,000–18,000 个中文字符的助眠叙事文稿。
- 用户一次提交，最终得到一个连续、可播放、只占一条 History 记录的 WAV。
- 内部过程文件在成功、失败或取消后及时清理，避免长期堆积冗余音频。
- 支持手填 `0.01–2.50`，保留两位小数，默认 `1.00`；`1.00` 完全旁路。
- 非 `1.00` 语速作用于最终完整音频，包括模型生成的句内停顿和应用插入的段间停顿。
- 全程本地处理；本轮不实现 iOS。

## 已确认的 Windows 行为

Windows 应用的核心 Python 逻辑被编译为 Cython `.pyd`，不存在可以可靠还原的原始 Python 源码。因此没有尝试“解密”或反编译后复制实现，而是结合二进制字符串、FFmpeg 参数和同稿 WAV 波形做黑盒确认。

确认结果：

- 切分目标约为 200 字，优先在中英文句末标点、逗号、分号或冒号附近结束。
- 生成的相邻片段之间先插入 0.3 秒零值静音。
- 所有片段和静音合并后，再对完整音频执行一次 `atempo=<速度>`。
- 因为停顿在变速前插入，速度为 `0.80` 时，0.3 秒停顿在最终文件中约为 `0.375` 秒。
- FFmpeg 通过管道或文件流处理，不要求把整条长音频一次性载入内存。

这些是常见的文本分段、PCM 拼接和时间伸缩行为。Vocello 以自己的 Swift 规划器、合并器、质量检查和文件生命周期重新实现，不包含 Windows 二进制或其代码表达。

## macOS 当前实现

| 环节 | 当前行为 |
| --- | --- |
| 路由 | 超过 200 个字符进入长文本路径 |
| 规划 | 复用 `SpokenTextPlanner` 和 `LongFormPlanner`，支持中日韩标点、受保护范围、稳定分段身份和 200 token 上限 |
| 生成 | 顺序、非流式逐段生成；逐段执行 Fast QC |
| 段间停顿 | 每个非末段边界固定插入 300 毫秒静音 |
| 合并 | `BoundedLongFormAssembler` 固定块读写，原子生成 24 kHz、单声道 PCM16 WAV |
| 变速 | 合并完成后，对完整 WAV 执行一次 FFmpeg `atempo`；超出单个滤镜范围时自动串联多个合法因子 |
| `1.00` | 不启动 FFmpeg、不重编码，保持原文件 |
| 发布 | 验证变速暂存 WAV 后原子替换；只保存一个最终 WAV 和一条 History |
| 清理 | History 接受后删除分段、失败片段、合并临时文件和变速暂存文件；失败或取消也清理任务工作区 |

当前规划器比 Windows 的简单正则切句更谨慎，会保护数字、版本号、URL 和缩写。产品目标是复刻可听见的分段长度、停顿和整段变速语义，不追求逐行复制未知实现。

## 为什么不是一次模型推理

Qwen3-TTS 的输出长度受 codec token、内存和生成稳定性限制。13,000–18,000 字远超适合一次请求处理的范围。单纯抬高 token 上限会集中放大 KV cache、尾段截断、随机漂移和失败重做成本。

因此正式语义是：

```text
完整文稿
  -> 约 200 字的标点感知规划
  -> 顺序生成与逐段质量检查
  -> 每段之间插入 0.3 秒静音
  -> 有界合并成一个 WAV
  -> 对完整 WAV 执行一次 FFmpeg atempo
  -> 验证并原子发布
  -> 写入一条 History
  -> 清理全部过程文件
```

## 语速算法决策

### 是否必须使用 FFmpeg

不必须。Apple `AVAudioUnitTimePitch` 和 FFmpeg `atempo` 都能实现“改变播放时长，同时尽量保持原音高”，因此从功能类别上看，macOS 可以完全使用系统框架完成保音高变速。

当前选择 FFmpeg 的原因不是 Apple 方案无法变速，而是产品目标包含“尽量复现已经投入生产使用的 Windows 版本听感”。在相同原始 WAV、相同文本和相同 `0.80`/`0.85` 速度下：

- Windows 修改版确认在完整音频合并后使用 FFmpeg `atempo`。
- Apple `AVAudioUnitTimePitch` 能正确延长时长并大体保持音高，但与 Windows 成品相比，声音包络、基频稳定性和频谱包络的偏差更大。
- FFmpeg `atempo` 的输出在实际试听中被确认与 Windows 版本效果一致。

因此需要区分两个目标：

| 产品目标 | Apple `AVAudioUnitTimePitch` | FFmpeg `atempo` |
| --- | --- | --- |
| 普通保音高变速 | 可以实现 | 可以实现 |
| 获得大体自然的 `0.80–0.90` 慢速语音 | 可以作为候选 | 可以实现 |
| 复现当前 Windows 助眠成品的听感 | 当前 A/B 差异明显 | 当前已通过试听确认 |
| 第三方组件与许可材料 | 不需要 | 需要 LGPL 合规、源码和许可材料 |
| App 体积 | 无额外二进制 | 增加约 1.28 MB 的最小 helper |
| 行为固定 | 依赖 Apple 系统实现 | 可固定 FFmpeg 版本、构建参数和滤镜参数 |

长文本能力本身不依赖 FFmpeg。文本分段、逐段生成、质量检查、静音插入、WAV 合并、过程文件清理和单条 History 发布均由 Vocello 自己完成；仅当用户选择非 `1.00` 速度时，最终完整 WAV 才进入调速处理器。`1.00` 始终旁路，不调用 FFmpeg，也不重新编码音频。

当前架构把 FFmpeg 做成独立 helper，而没有把 FFmpeg 库链接进 Swift 主程序。这不仅明确了许可证和签名边界，也保留了替换空间：以后若 Apple TimePitch 的参数调优、新系统实现或其他原生算法在盲听和客观指标上达到目标，只需替换最终调速处理器，不需要重写长文本生成链路。

后续只有满足以下条件时，才重新考虑用 Apple TimePitch 替换 FFmpeg：

1. 使用相同原始 WAV、相同速度和相同停顿语义进行同源 A/B，不能只比较不同次模型生成的音频。
2. 对 `0.80`、`0.85`、`0.90` 三个常用速度分别进行盲听，至少达到当前 `atempo` 成品的可接受程度。
3. 重新检查声音包络、基频、频谱包络、总时长和段间停顿，不得用单一指标代替完整质量判断。
4. 验证长达数小时的最终 WAV 可以有界处理，失败、取消和磁盘空间不足时仍能正确清理临时文件。
5. 明确同一个速度值不会因为系统版本或处理路径不同而向用户输出两种明显不同的声音。

在这些条件完成以前，macOS 正式路径继续使用固定版本的 FFmpeg `atempo`；Apple TimePitch 不作为静默回退，避免同一速度值产生不可预测的音质变化。

### 已否决：改采样率或普通 Varispeed

该方法会同时改变速度和音高，无法保持克隆音色身份，不作为正式产品方案。

### 已否决：Apple `AVAudioUnitTimePitch`

Apple 方案不增加第三方许可证，但同源 A/B 显示听感和客观指标都明显弱于 FFmpeg `atempo`：波形包络相关性更低，基频和频谱包络误差更大。它不再作为当前回退路径，以免用户在同一个速度值下得到两种明显不同的质量。

### 当前开发路径：FFmpeg `atempo`

- 常用速度如 `0.80`、`0.85`、`0.90` 使用单个 `atempo`。
- `atempo` 单级因子限定在 `0.5–2.0`；`0.01–2.50` 的极端输入由多个合法因子相乘实现。
- 处理对象是已经合并并插入停顿的最终 WAV，因此整体时长约为原时长除以速度值。
- FFmpeg 写入隐藏暂存 WAV，应用验证采样率、声道、位深、帧数与可读性后再原子替换原成品。
- `1.00` 完全旁路，不因启用语速功能而产生无意义重编码。

## 停顿与时长语义

固定 0.3 秒静音在完整音频变速前插入，因此最终停顿也会按速度伸缩：

| 速度 | 0.3 秒段间停顿的最终时长 |
| ---: | ---: |
| `1.00` | 0.300 秒 |
| `0.90` | 约 0.333 秒 |
| `0.85` | 约 0.353 秒 |
| `0.80` | 0.375 秒 |

这与已分析的 Windows 输出一致，也解释了它比只拉伸发声区域的方案听起来更舒缓。新任务默认 `1.00`，不会静默继承上一次输入。

## 临时文件与恢复边界

- 分段 WAV 只存在于任务私有工作区，不作为用户资产展示。
- 合并器和 FFmpeg 都使用临时目标；最终文件通过验证后原子发布。
- 成功任务最终只保留一个 WAV 和必要的小型摘要元数据。
- 失败、取消和被替换的片段及时删除；启动时清扫无法对应有效任务的孤儿目录。
- 当前版本没有断点恢复，失败后整篇重试。以后若增加恢复，缓存身份必须绑定文本摘要、模型、音色、采样参数、分段算法、停顿和语速算法版本。

## 分发与许可证边界

普通 Xcode 开发构建仍可解析本机固定位置中已经存在的 FFmpeg，用于快速开发。正式发布路径不会复制该文件，而是从固定 FFmpeg 8.0.3 官方源码构建独立 `Contents/Helpers/ffmpeg-vocello`。组件只开放本地 file、WAV、PCM16 和 `atempo` 所需能力，明确禁用 GPL、version3、nonfree、网络和外部库自动发现。

正式组件现已具备：

1. 固定源码、分离签名、摘要、configure 参数和完整能力白名单。
2. 以独立进程隔离 Vocello 主程序与 FFmpeg，并把完整 LGPL、上游 LICENSE、NOTICE 和构建身份放入 App。
3. 在正式发布时单独签名 helper，检查 Hardened Runtime、Team ID 和包内内容。
4. 把精确对应源码、分离签名和构建身份作为 GitHub Release 资产，并纳入 SBOM、发布证据和校验和。
5. 保持普通法律审查边界；完整说明见[macOS FFmpeg LGPL-only 正式发布组件](ffmpeg-lgpl-release-component.md)。

独立复刻通用行为降低了复制未知代码表达的版权风险，但不自动免除 FFmpeg 许可证、上游模型许可证、商标和 App Store 政策义务。

## 确定性验收

- 200 字级规划应完整覆盖原文、顺序不变、无丢字，并保持受保护范围。
- 所有非末段边界的计划停顿均为 300 毫秒，末段为 0。
- `atempo` 链中每个因子均落在 `0.5–2.0`，乘积等于规范化后的用户速度。
- `1.00` 即使 FFmpeg 不存在也必须成功旁路。
- 非 `1.00` 输出必须为可读的 24 kHz、单声道 PCM16 WAV，时长落在容差内。
- 变速失败不得覆盖原成品；成功、失败和取消都不得遗留隐藏暂存 WAV。
- 13,000–18,000 字应只产生一个最终 History 项目，过程文件在接受后清理。
