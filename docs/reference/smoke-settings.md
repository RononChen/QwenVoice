# Smoke Runbook: Settings surface renders correctly

Lightweight functional smoke for the Settings screen: the screen mounts, the recommended-models status renders, at least one Custom Voice model package shows as "Ready".

Follows the [Standard smoke skeleton](ui-test-surface.md#standard-smoke-skeleton) for setup + teardown. This file documents the Settings-specific drive sequence and verify steps.

## Mode-specific inputs

| Field | Value |
|---|---|
| Section to land on | top of Settings → Model downloads |
| Model id asserted | `pro_custom` (Custom Voice) |
| Expected status | At least one Custom Voice package row, such as `Lite (0.6B 4-bit)` or `Speed (1.7B 4-bit)`, shows "Ready" |
| smoke-check arg | `custom` (without it Settings would show "not installed" instead of "Ready") |

## Mode-specific deltas

Skeleton Phases 1–3 are unchanged. In Phase 4:

1. **Navigate to Settings**: click `sidebar_settings`. Confirm with `scripts/uitest.sh locate screen_settings` (exit 0).
2. **Verify the Custom Voice model package renders**:
   - `scripts/uitest.sh locate settings_packageStatus_pro_custom` should succeed.
   - `scripts/uitest.sh locate settings_package_pro_custom` should also succeed.
   - Cross-check via screenshot — the section shows Lite/Speed/Quality package rows for the variants declared in `qwenvoice_contract.json`; at least one is labeled "Ready".
3. **Optional variant-package check**: the Lite row exposes `settings_packageStatus_pro_custom_compact_speed` when installed (or `…_speed` / `…_quality` for other variants). If the AX id doesn't resolve, screenshot and read the visible "Ready" / "Repair" / "Download" badge per row.

### Skipping `verify-generation`

This smoke doesn't generate audio. In Phase 5, write `result.json` manually:

```sh
cat > "$ART/result.json" <<JSON
{
  "pass": true,
  "screen": "settings",
  "installed_variants": [<labels visible as Ready>],
  "timestamp": "$(date -u +%Y-%m-%dT%H:%M:%SZ)"
}
JSON
```

Set `"pass": false` and add `"reason"` if `settings_packageStatus_pro_custom` doesn't resolve or no variant shows "Ready".

## What this smoke will NOT do

- Will not trigger a model download (multi-GB; would skew the artifact dir and burn network).
- Will not exercise the repair/delete/manage flows. Out of scope for the happy-path smoke.
- Will not modify any installed models or saved voices.

## Failure handling

- **Settings screen doesn't mount**: `locate screen_settings` fails. Take a screenshot — did the sidebar click land on something else? Retry once with a fresh `prep`.
- **No models show as Ready**: the `smoke-check custom` prerequisite was probably stale (or models were deleted between the check and the run). Re-run `smoke-check custom` outside the runbook to confirm; if it still passes there's likely a UI rendering bug — capture the screenshot and report.
- Perceptual review doesn't apply (no audio generated by this smoke).
